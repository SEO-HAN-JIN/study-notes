## 2. 작업 시작할 때 하는 일

### 2-1. 터미널 열기

Mac 기준으로 터미널을 연다.

---

### 2-2. 프로젝트 폴더로 이동

내 MkDocs 프로젝트 폴더로 이동한다.

```bash
cd study-notes
```
여기서 study-notes는 내가 만든 프로젝트 폴더명이다.

--- 

### 2-3. Python 가상환경 활성화

MkDocs와 관련 패키지를 가상환경 안에 설치했기 때문에
작업할 때는 먼저 가상환경을 활성화해야 한다.

```bash
source .venv/bin/activate
```

정상적으로 활성화되면 터미널 앞에 보통 `(.venv)`가 붙는다.

예시:

```bash
(.venv) seohanjin@MacBookPro study-notes %
```

## 3. 글 작성 또는 수정
### 3-1. VSCode에서 프로젝트 열기

VSCode에서 `study-notes` 폴더를 연다.

---

### 3-2. 수정할 Markdown 파일 열기

예를 들어 아래와 같은 파일을 수정할 수 있다.

- `docs/index.md`
- `docs/certifications/cka/index.md`
- `docs/certifications/cka/week1/theory.md`
- `docs/certifications/cka/week1/wrong-answers.md`

즉, `docs/` 폴더 안의 Markdown 파일을 수정하면 된다.

---

### 3-3. 새 글을 추가하는 경우

새 주제를 만들고 싶으면 파일을 추가한다.

예시:
```bash
mkdir -p docs/certifications/cka/week7
touch docs/certifications/cka/week7/theory.md
touch docs/certifications/cka/week7/practice.md
touch docs/certifications/cka/week7/wrong-answers.md
```

그리고 `mkdocs.yml`의 `nav`에도 새 파일을 연결해준다.

예시:

```yaml
- CKA:
    - 소개: certifications/cka/index.md
    - Week 7:
        - 개념정리: certifications/cka/week7/theory.md
        - 문제풀이: certifications/cka/week7/practice.md
        - 오답노트: certifications/cka/week7/wrong-answers.md
```

## 4. 로컬에서 미리 보기

글을 수정한 뒤에는 바로 GitHub에 올리지 말고
먼저 내 컴퓨터에서 사이트가 잘 보이는지 확인하는 것이 좋다.

```bash
mkdocs serve
```

이 명령어를 실행하면 로컬 개발 서버가 실행된다.

보통 브라우저에서 아래와 비슷한 주소로 접속하면 된다.

```bash
http://127.0.0.1:8000/
```

- 이 단계에서 확인할 것
- 새 글이 메뉴에 잘 보이는지
- Markdown 문법이 깨지지 않았는지
- 코드 블록이 잘 표시되는지
- 한글이 정상적으로 보이는지
- `mkdocs.yml` 문법 오류가 없는지

--- 
## 5. GitHub 원본 저장소에 반영
로컬에서 확인 후 문제가 없으면 GitHub에 원본을 올린다.

### 5-1. 변경 파일 확인
```bash
git status
```
현재 수정된 파일 목록을 확인할 수 있다.

---

### 5-2. 변경 파일 추가
```bash
git add .
```
모든 변경 파일을 스테이징한다.
특정 파일만 올리고 싶다면 파일명을 지정해도 된다.

예시:
```bash
git add docs/certifications/cka/week7/theory.md
git add mkdocs.yml
```

---

### 5-3. 커밋
```bash
git commit -m "add cka week7 notes"
```

커밋 메시지는 작업 내용을 알아볼 수 있게 작성하는 것이 좋다.

예시:

- `update cka week6 wrong answers`
- `add aws saa networking notes`
- `fix mkdocs nav`
- `add blog draft for ingress`

---

### 5-4. GitHub로 push
```bash
git push
```

이제 GitHub 저장소의 main 브랜치에 원본 Markdown 파일들이 반영된다.

---

## 6. GitHub Pages 사이트에 재배포

GitHub 원본 저장소에 올렸다고 해서
바로 웹사이트가 자동으로 바뀌는 것은 아니다.

MkDocs로 GitHub Pages용 브랜치(`gh-pages`)에 다시 배포해야 한다.

```bash
mkdocs gh-deploy --force
```

이 명령어를 실행하면:

- 현재 `docs/` 내용 기준으로 사이트를 다시 빌드하고
- `gh-pages` 브랜치에 업로드해서
- GitHub Pages 사이트를 갱신한다

즉,
- `git push` = 원본 저장소 업데이트
- `mkdocs gh-deploy --force` = 공개 사이트 업데이트

이 두 개는 역할이 다르다.

---

## 7. 사이트에서 최종 확인

배포가 끝나면 내 GitHub Pages 주소로 접속해서 확인한다.

예시:
```bash
https://SEO-HAN-JIN.github.io/study-notes/
```

여기서 확인할 것:

- 새 글이 반영되었는지
- 메뉴가 잘 보이는지
- 링크가 깨지지 않았는지
- 오타나 문법 문제가 없는지

---

## 8. 작업 종료

작업이 끝났으면 가상환경을 종료할 수 있다.
```bash
deactivate
```

꼭 종료해야 하는 것은 아니지만, 정리 차원에서 끄고 끝내도 된다.

## 10. 가장 자주 하는 실수
### 10-1. 가상환경 활성화를 안 함

증상:
```bash
zsh: command not found: mkdocs
```

원인:
- `.venv` 활성화를 안 해서 mkdocs 명령을 못 찾는 경우

해결:
```bash
source .venv/bin/activate
```

---

### 10-2. `git push`만 하고 사이트가 안 바뀐다고 생각함

원인:
- GitHub 저장소 원본만 업데이트하고
- `mkdocs gh-deploy --force`를 안 해서 Pages는 갱신되지 않은 경우

해결:
```bash
mkdocs gh-deploy --force
```

--- 

### 10-3. 새 파일을 만들었는데 메뉴에 안 보임

원인:
- `mkdocs.yml`의 nav에 연결 안 한 경우

해결:
- `mkdocs.yml`에 해당 Markdown 파일 경로 추가

---

### 10-4. YAML 들여쓰기 오류

증상:
- `mkdocs serve` 또는 `mkdocs gh-deploy` 실행 시 오류 발생

원인:
- `mkdocs.yml` 들여쓰기 문제

해결:
공백 개수와 구조를 다시 확인

---
